<objective>
Wire Unkey-powered API key authentication, distributed rate limiting, and rate limit response headers into the OraClaw public API, replacing the current prefix-based auth stubs and in-memory rate counters.

Purpose: The current auth system uses string prefix matching (ok_test_* = starter) and an in-memory Map for rate limiting that resets on every deploy. This plan replaces it with Unkey's verifyKey() which handles authentication, tier resolution, and rate limiting in a single API call with edge-distributed persistence.

Output: Three new/modified files -- Unkey client singleton, auth middleware with rate limit headers, and cleaned-up api-public.ts with old auth code removed.
</objective>

**Create `mission-control/apps/api/src/services/unkey.ts`:**

1. Import `Unkey` from `@unkey/api` (class-based SDK v2.3.2, NOT functional imports).
2. Export a singleton `unkey` instance: `new Unkey({ rootKey: process.env.UNKEY_ROOT_KEY! })`.
3. Export `TIER_RATE_LIMITS` config object mapping tier names to `{ limit, duration }`:
   - starter: `{ limit: 1667, duration: 86400000 }` (~50K/month as daily)
   - growth: `{ limit: 16667, duration: 86400000 }` (~500K/month)
   - scale: `{ limit: 166667, duration: 86400000 }` (~5M/month)
   - enterprise: `{ limit: 1000000, duration: 86400000 }` (effectively unlimited)
4. Export `createApiKey(tier: string, stripeCustomerId: string, userEmail: string)` function that calls `unkey.keys.createKey()` with:
   - `apiId: process.env.UNKEY_API_ID!`
   - `prefix: 'ok_live'`
   - `name: \`${userEmail} - ${tier}\``
   - `externalId: stripeCustomerId`
   - `meta: { tier, stripeCustomerId }`
   - `ratelimits: [{ name: 'api_calls', ...TIER_RATE_LIMITS[tier], autoApply: true }]`
   - `enabled: true`
   - Return `data` (contains `key` and `keyId`), throw on `error`.
5. Export `rotateApiKey(keyId: string, expirationMs: number = 3600000)` that calls `unkey.keys.rerollKey({ keyId, expiration: expirationMs })`. Return new key data, throw on error.
6. Export `revokeApiKey(keyId: string)` that calls `unkey.keys.deleteKey({ keyId })`. Throw on error.

**Create `mission-control/apps/api/src/middleware/auth.ts`:**

1. Add Fastify module augmentation declaring custom request properties:
   ```typescript
   declare module 'fastify' {
     interface FastifyRequest {
       tier: string;
       keyId?: string;
       stripeCustomerId?: string;
       rateLimitRemaining?: number;
       rateLimitLimit?: number;
       rateLimitReset?: number;
       billingPath: 'stripe' | 'free';
     }
   }
   ```
2. Export `createAuthMiddleware(unkeyClient: Unkey)` that returns a Fastify `preHandler` async function:
   - Check `request.headers.authorization`. If missing: set `request.tier = 'free'`, `request.billingPath = 'free'`, return (skip Unkey).
   - Extract key: `authHeader.replace('Bearer ', '')`.
   - Call `unkeyClient.keys.verifyKey({ key })`.
   - If `error` (SDK/network error): log error, return 503 with RFC 9457-style body `{ type, title, status, detail }`.
   - Extract rate limit info from `data.ratelimits?.[0]`. If present, set headers on `reply` directly AND store on `request` for the onSend hook.
   - If `!data.valid`: map `data.code` to HTTP status (NOT_FOUND->401, RATE_LIMITED->429, DISABLED->403, EXPIRED->401, USAGE_EXCEEDED->429, FORBIDDEN->403), return error with RFC 9457-style body.
   - If valid: set `request.tier`, `request.keyId`, `request.stripeCustomerId`, `request.billingPath = 'stripe'`.
3. Export `rateLimitHeadersHook` as an `onSend` async hook that checks `request.rateLimitRemaining !== undefined` and sets X-RateLimit-Remaining, X-RateLimit-Limit, X-RateLimit-Reset headers on the reply. This ensures headers appear on ALL responses including successful ones (the preHandler sets them on errors, onSend catches success paths).
4. Use `fastify.decorateRequest()` in a plugin wrapper to declare the custom properties with default values (tier='free', billingPath='free', others undefined) so Fastify's serialization works correctly.

**IMPORTANT:** Do NOT use `@unkey/ratelimit` package. The `verifyKey()` with `autoApply: true` handles rate limiting. Do NOT create a separate rate limit check -- it is built into the single `verifyKey()` call.
